## Purpose

This document combines the PDF review findings with the additional ATS/WCAG feedback and turns them into a production-ready fix spec.

The goal is to produce a resume PDF that is:

- ATS-safe
- screen-reader navigable
- PDF accessibility compliant enough to support a WCAG/PDF-UA review
- credible for a resume that explicitly claims accessibility, WCAG 2.2, design systems, and screen-reader testing experience

# 1. Correct the Core Assumption

## Accessibility Tags Are Mandatory for Accessibility, Not Strictly Mandatory for ATS

The added feedback is directionally right but overstates one point.

Semantic PDF tags are mandatory for accessibility because assistive technologies use the PDF structure tree to understand headings, paragraphs, lists, links, and reading order.

For ATS, the situation is different. Most ATS parsers do not meaningfully evaluate PDF accessibility tags such as `<H1>`, `<H2>`, or `<L>`. They usually extract the text layer and then classify the content using text patterns, section names, dates, and job/entity proximity.

So the correct position is:

> Good PDF tagging supports accessibility directly and can support ATS indirectly by preserving clean reading order. But ATS optimization still depends mostly on simple layout, standard section names, consistent dates, and clean text extraction.

Do not tag the PDF as a magic ATS hack. Tag it because the resume should be accessible and because good structure reduces parsing risk.

# 4. Fix Job Entity Grouping

## Problem

The current job metadata is visually clear, but ATS parsing can become fragile when job title, company, location, and date are separated without strong grouping.

Current pattern:

```text
Principal Product Designer & Strategist
UX Design Lab, LLC | City, State 04/2024 - Present
```

This is acceptable visually, but the final PDF must ensure the reading order keeps each job title directly tied to its company and date.

## Recommended Pattern

Use this pattern consistently:

```text
Principal Product Designer & Strategist
UX Design Lab, LLC | City, State | Apr 2024 - Present
```

Then follow immediately with bullets.

## Rules

- Keep job title on one line.
- Keep company, location, and date on the next line.
- Use the same separator every time: ` | `.
- Do not float dates in a separate right-aligned text box unless the source tool preserves reading order perfectly.
- If right-aligning dates visually, ensure the tag order still reads: title, company, location, date, bullets.

Best option for ATS: keep dates inline with company/location instead of floating them to the far right.

# 5. Standardize Date Formatting

## Problem

The current PDF mixes date formats:

```text
04/2024 - Present
01/2022 - 04/2024
04.2013 - 08.2017
```

This is sloppy and creates parser risk.

## Fix

Use month abbreviation + year everywhere:

```text
Apr 2024 - Present
Jan 2022 - Apr 2024
Jan 2020 - Nov 2021
Oct 2017 - Jan 2020
Apr 2013 - Aug 2017
```

## Rules

- Use three-letter month names.
- Use four-digit years.
- Use spaces around the hyphen.
- Use `Present`, not `Current`.
- Use one format everywhere.

This is more ATS-friendly than numeric dates because it avoids ambiguity between U.S. and international formats.

# 6. Convert Responsibility Text Into Real Lists

## Problem

Some extracted bullets behave like plain paragraphs. That is not ideal.

For accessibility, list content must be tagged as lists.

For ATS, visible bullet delimiters help separate accomplishments.

## Fix

Every achievement/responsibility item should be a real list item in the source file, not a manually styled paragraph.

Use standard bullets:

```text
• Embedded accessibility into the design system architecture, component rules, and review workflow, reducing critical WCAG defects by 18%.
• Directed the integration of Figma components with Storybook and AEM, reducing production time by 40% and vendor dependency costs by 50%.
```

## PDF Tags Required

Each bullet list must be tagged as:

```text
<L>
  <LI>
    <Lbl>•</Lbl>
    <LBody>Bullet text...</LBody>
  </LI>
</L>
```

## Rules

- Use real list styles in Word, InDesign, or the authoring system.
- Do not create bullets with manually typed symbols unless the PDF export still tags them correctly.
- Keep bullets concise.
- Lead with outcome where possible.
- Do not bury metrics at the end of long paragraphs.

# 11. Do Not Export the Final Accessible Resume From Chrome Alone

## Problem

The current PDF was produced by HeadlessChrome / Skia. This workflow can produce a visually clean PDF, but it does not reliably produce a tagged accessible PDF.

## Better Authoring Options

Use one of these workflows:

### Option A: Microsoft Word → Tagged PDF → Acrobat Pro

Best practical workflow for this resume.

Steps:

1. Build the resume in Word.
2. Use real Word styles:
   - Title or Heading 1 for the name.
   - Heading 2 for major sections.
   - Heading 3 for job titles and degree titles.
   - Normal style for paragraphs.
   - Real list styles for bullets.
3. Export as PDF using accessibility/tagging options enabled.
4. Open the PDF in Acrobat Pro.
5. Run accessibility checks.
6. Repair tags, links, reading order, and metadata.

### Option B: InDesign → Tagged PDF → Acrobat Pro

Acceptable if the designer knows tagged PDF export.

Steps:

1. Use paragraph styles.
2. Map styles to PDF tags.
3. Anchor content in correct story order.
4. Avoid disconnected text boxes unless reading order is controlled.
5. Export with `Create Tagged PDF` enabled.
6. Verify in Acrobat Pro.

### Option C: HTML/CSS → PDF With Accessibility-Aware Renderer

Only use this if the renderer supports accessible tagged PDF output.

Do not assume browser print-to-PDF is enough.

If using HTML, the HTML must have:

```html
<h1>Jane Doe</h1>
<h2>Professional Summary</h2>
<h2>Key Achievements</h2>
<h2>Professional Experience</h2>
<h3>Principal Product Designer & Strategist</h3>
<ul>
  <li>...</li>
</ul>
```

But the PDF renderer still must preserve those semantics into PDF tags.

# 13. ATS-Safe Formatting Rules

## Use Standard Resume Section Names

Preferred:

```text
Professional Summary
Key Achievements
Professional Experience
Skills
Awards and Recognition
Education
```

Avoid overly branded headers like:

```text
Selected Impact
What I Bring
The Work
Proof
Where I Made Things Better
```

That kind of language may be fine on a portfolio. In a resume, it is noise.

## Avoid Layout Risk

Do not use:

- Tables for layout.
- Multi-column resume structure.
- Text boxes that disrupt reading order.
- Icons without text labels.
- Infographics.
- Skill bars.
- Sidebars.
- Hidden text.
- White text keyword stuffing.

Use:

- One-column layout.
- Standard headings.
- Plain text contact info.
- Real bullets.
- Consistent job blocks.
- Consistent date formatting.

## ATS Text Extraction Test

Before sending the resume, run these checks:

1. Open the PDF.
2. Select all text.
3. Copy and paste into a plain text editor.
4. Confirm the pasted text reads in the correct order.
5. Confirm each role keeps its company and dates.
6. Confirm bullets remain separated.
7. Confirm education does not collapse into nonsense.
8. Confirm contact information appears near the top.

Optional command-line tests:

```bash
pdftotext resume.pdf resume.txt
pdftotext -layout resume.pdf resume-layout.txt
```

Then inspect both text files.

The non-layout version matters because many parsers do not preserve visual layout.
